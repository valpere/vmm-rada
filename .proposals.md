## A. Agent evolution path: from static pipeline to dynamic router

After studying Go AI agent frameworks (`vitalii-honchar-go-agent`, `SwarmGo`,
`LangChainGo`, `Eino`, Google ADK), the most natural evolution for vmm-rada's
3-stage pipeline is:

- **Stage 1 → 3** maps directly to a linear agent chain
- **Stage 2's label→model mapping** is the seed of a routing table — exactly what
  SwarmGo's agent transitions model
- **The Chairman** is already acting as a meta-agent (synthesising other agents' outputs)

The `Runner` interface is the migration path. When we want to swap the static pipeline
for an agent-based implementation, we do it behind `Runner` without touching the handler.

**Status:** Architecture in place. No action needed until we decide to experiment with
an agent framework backend.

## B. FPF alignment: vmm-rada embodies first-principles thinking naturally

The First Principles Framework (FPF) describes a "Wide search before selection" pattern.
The 3-stage council pipeline IS this pattern:

- Stage 1 = wide search (diverse LLM responses without anchoring)
- Stage 2 = structured evaluation (independent ranking, no groupthink)
- Stage 3 = synthesis by a meta-agent informed by the full evidence set

FPF's "Transformer quartet" (Agent, Method, Description, Work) maps to our pipeline:
- Agent = council models (S1), rankers (S2), Chairman (S3)
- Method = prompt templates (`prompts.go`)
- Description = `StageOneResult`, `StageTwoResult`, `StageThreeResult`
- Work = the actual LLM response content

**Future idea:** `CouncilConfig` with model metadata (provider, context window, cost
tier) so the chairman prompt can reference "Response A came from a model with X context
window" — increasing synthesis transparency.

## E. Circuit breaker for OpenRouter client

When OpenRouter is down, all 4 council models fail but we wait in parallel (120s total).
A circuit breaker decorator on `LLMClient` would fail fast after N failures. Since
`LLMClient` is an interface, this is a pure decorator — no changes to `Rada`.

**Priority:** low until production load. Create a p3 issue when the need arises.
