# VMM Rada — User Guide

VMM Rada is a deliberation tool that asks several AI models the same question, has them peer-review each other's answers, then produces a synthesized final answer. The goal is a response that is more considered, more balanced, and less prone to a single model's blind spots than any individual model could produce alone.

---

## Getting Started

### Prerequisites

- The **Go backend** must be running before you open the interface. By default it runs on `http://localhost:8001`.
- The **frontend dev server** must be running: `npm run dev` (starts at `http://localhost:5173`).

### Opening the app

Navigate to `http://localhost:5173` in your browser. You will see the VMM Rada interface — a sidebar on the left and a chat area on the right.

**Sidebar** — lists your saved conversations. The title of each conversation is auto-generated by the backend once the pipeline finishes. Click any conversation to reload it.

**Chat area** — shows the question and the full three-stage response. When no conversation is selected, it shows a welcome heading ("Welcome to VMM Rada") and a prompt to create a new conversation ("Create a new conversation to get started").

---

## Creating a Conversation

1. Click **+ New Conversation** in the sidebar header.
2. A blank conversation opens in the chat area.
3. Type your question in the text area.
4. Press **Enter** to send, or **Shift+Enter** to add a line break within your question.
5. Alternatively, click the **Send** button.

> **One question per conversation.** Each conversation holds exactly one question and one response. The input field disappears after you send. To ask another question, create a new conversation.

---

## The Three-Stage Pipeline

After you send your question, the backend runs a three-stage deliberation pipeline. The interface updates progressively as each stage completes — you do not need to wait for the whole pipeline to finish.

### Stage 1 — Individual Responses

Each council model answers your question independently, without seeing what the others said.

The interface shows a spinner ("Running Stage 1: Collecting individual responses...") while the models are working. When Stage 1 completes, a tabbed panel appears:

- Each tab shows one model's name (shortened to the model ID, e.g. `gpt-5.1`).
- Click a tab to read that model's full answer.
- The full model identifier (e.g. `openai/gpt-5.1`) is shown above the response text.
- Responses are rendered as Markdown — headers, bullet points, and code blocks are formatted.

**What to look for:** read a few tabs to see where the models agree, where they diverge, and which answers feel most authoritative to you. This sets context for understanding Stage 2.

### Stage 2 — Peer Rankings

Each model re-reads all responses (anonymized as "Response A", "Response B", etc.) and ranks them from best to worst, with written justification.

The interface shows a spinner ("Running Stage 2: Peer rankings...") while the evaluations are processed. When Stage 2 completes:

#### Raw Evaluations (tabs)

Each tab shows one model's evaluation of all the other responses. The anonymous labels ("Response A", "Response B", …) are replaced with **bold model names** for readability. Below the written evaluation, an "Extracted Ranking" shows that model's final ordered preference list.

> The description in the UI explains that the original evaluation used anonymous labels — model names are shown in bold only as a display convenience.

#### Aggregate Rankings ("Street Cred")

Below the individual evaluations is a summary table that combines all votes into a single ranked list:

- **#1, #2, #3, …** — overall position
- **Model name** — the model being ranked
- **Avg:** — average rank position across all evaluators (lower is better; 1.0 would mean every evaluator ranked it first)
- **(N votes)** — how many evaluations contributed

This table answers: "Which response did the council collectively judge to be best?"

### Stage 3 — Final Rada Answer

The Chairman model reads the question, all Stage 1 responses, and the Stage 2 rankings, then synthesizes a single final answer.

The interface shows a spinner ("Running Stage 3: Final synthesis...") while the Chairman is working. When Stage 3 completes, the final answer appears:

- **Chairman: \<model-name\>** — the model acting as synthesizer
- The response is rendered as Markdown

The Chairman's job is not to pick a winner but to integrate the best elements from all responses, acknowledge disagreements, and produce a more complete answer than any single model could.

**If Stage 3 fails** (e.g. all council models returned errors), an error banner appears instead:
```
⚠  <error message>
```

## Live Progress Indicators

While the pipeline runs, you will see several indicators:

| Indicator | Meaning |
|-----------|---------|
| "Consulting the council..." (bottom spinner) | The overall pipeline is active |
| "Running Stage 1: Collecting individual responses..." | Stage 1 in progress |
| "Running Stage 2: Peer rankings..." | Stage 2 in progress |
| "Running Stage 3: Final synthesis..." | Stage 3 in progress |

Stages appear in order; each spinner disappears when that stage's results arrive.

---

## Conversation History

The sidebar shows all conversations, newest first. Each entry shows:
- **Title** — auto-generated by the backend based on your question (updated when the pipeline finishes). Shows "New Conversation" until the title arrives.
- **N messages** — the message count (always 2 for a complete exchange: your question + the council response).

Click any conversation to reload it. Previously completed conversations reload fully from the backend. Note that **Stage 2 de-anonymization is not preserved** — when reloading a saved conversation, the Stage 2 "Raw Evaluations" will show the original anonymous labels ("Response A", "Response B", …) rather than bold model names, because the label mapping is ephemeral and not stored.

## Tips

**Ask open questions.** The pipeline is most valuable for questions where there is genuine room for multiple perspectives — analysis, comparison, explanation, trade-offs. Factual yes/no questions produce less interesting disagreement.

**Read Stage 2 before Stage 3.** The aggregate ranking tells you which model the council itself found most convincing. Compare that to your own judgement from Stage 1. If the top-ranked model's answer surprised you, Stage 2's written evaluations explain why the council preferred it.

**Check the Chairman's model.** Stage 3's Chairman is configurable on the backend (`CHAIRMAN_MODEL` env var). Different models synthesize differently — some are more conservative, some more opinionated.

**One question = one conversation.** If you want a follow-up, create a new conversation. The pipeline does not support multi-turn exchanges; each question is treated as a standalone deliberation.

---

## Limitations

| Limitation | Detail |
|-----------|--------|
| One question per conversation | The input disappears after sending; create a new conversation to ask something else |
| No multi-turn conversation | The pipeline always starts from scratch with the full question; prior context is not fed back |
| Stage 2 de-anonymization is ephemeral | Reloaded conversations show anonymous labels in Stage 2, not model names |
| No question editing | There is no way to edit a sent question; start a new conversation |
| No response regeneration | There is no retry or regenerate button |

---

## Backend Configuration (for reference)

The backend controls which models participate. Relevant environment variables:

| Variable | Purpose |
|----------|---------|
| `RADA_MODELS` | Comma-separated list of models used in Stage 1 and Stage 2 |
| `CHAIRMAN_MODEL` | The model used for Stage 3 synthesis |
| `TITLE_MODEL` | The model used to generate conversation titles (runs concurrently with the pipeline) |
| `PORT` | Backend port (default `8001`) |

The frontend connects to the backend at the URL set by `VITE_API_BASE` (default `http://localhost:8001`). See `.env.example`.
